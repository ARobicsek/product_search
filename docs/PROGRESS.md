**Finding worth remembering:** The "(no description)" Flags bug had been visible in every live report for months, but read as either expected ("subscription doesn't need a description") or cosmetic-only. Once root-caused, it was a 1-line key swap to fix `low_feedback` — but the broader truth is that the fallback dict's *keying convention* was ambiguous (rule names vs flag labels) and the renderer assumed only one. A symptom-first patch (key swap) would have left every future user-chosen flag label vulnerable. The right move was to make the lookup robust to *both* identifiers a flag carries — and stop rendering the misleading literal placeholder when nothing resolves. Generalises: when a fallback table looks up by an externally-controlled key, the lookup needs to be tolerant of the multiple identifiers that key might map through, not assume a single canonical form.
