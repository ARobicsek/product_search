## Current state — 2026-05-25 onboarder paper-cut — `description:` schema-vs-onboarder gap (ADR-092)

**Deliverables:** ADR-092 in DECISIONS.md. Schema edits: `description: str = ""` in [worker/src/product_search/profile.py](../worker/src/product_search/profile.py); matching optional path in [web/lib/onboard/schema.ts](../web/lib/onboard/schema.ts) (`description?: string` + guarded validator). `display_name` fallback in [worker/src/product_search/validators/ai_filter.py](../worker/src/product_search/validators/ai_filter.py). Prompt edit in [worker/src/product_search/onboarding/prompts/onboard_v1.txt](../worker/src/product_search/onboarding/prompts/onboard_v1.txt) (SHOULD-emit comment with rationale) regen'd into `promptText.ts` via `sync-prompt.js`. 2 new worker tests.

**What shipped:**
- **Schema — description is now optional in both layers.** `description` defaults to `""` in the worker `Profile` model; the web YAML validator no longer fails when the key is absent. A wrong-typed value (number, array) still errors — only the *missing/null* case is permitted. The save round-trip the user just experienced ("profile failed schema validation / description: expected string" on the "The Week 1yr subscription" draft) is no longer reachable from here.
- **Runtime — ai_filter falls back to display_name.** When `description` is empty or missing, [ai_filter.py:198](../worker/src/product_search/validators/ai_filter.py#L198) reads `display_name` instead, so the system prompt's "Description: …" line is always meaningful and never blank. `display_name` is already populated for every profile and is what the rest of the system uses; this is the lowest-cost fallback that preserves prompt readability.
- **Prompt — SHOULD-emit callout.** Added a 5-line comment next to the `description:` line in the schema template that says the field is optional but the model SHOULD emit it because the AI filter reads it for context beyond `display_name`. The trade-off is named (recall benefit) rather than the rule restated ("required"), because the previous "required" framing did not prevent omission.
- Green: worker pytest 353/353 (+2: `test_description_optional_when_omitted` pins schema acceptance + `""` default; `test_system_prompt_falls_back_to_display_name_when_description_empty` captures the ai_filter `system=` kwarg and pins both "Description: {display_name}" present + "Description: \n" absent). Web tsc 0 errors; eslint 4 pre-existing warnings unchanged; `test:guards` 11/11; `test:parity` 2/2. `sync-prompt.js` produced the expected single-line `promptText.ts` delta only.

**Finding worth remembering:** Defense-in-depth at *both* the schema layer and the prompt layer is appropriate when (a) the failing field is **not load-bearing** (the runtime works perfectly fine with `description=""`) AND (b) the failure has been observed in the wild. The schema fix removes the failure mode entirely (it's now unreachable); the prompt fix preserves the upside (richer AI-filter context) without making it a hard requirement. The PROGRESS.md candidate framing — "optional-with-default OR always-emit from the prompt" — presented these as alternatives; in retrospect they're complements with different roles (schema = floor, prompt = ceiling). Pattern: when an onboarder paper-cut is rooted in an "LLM omitted a field" miss, ask first "is this field load-bearing?" — if no, fix the data layer first.
