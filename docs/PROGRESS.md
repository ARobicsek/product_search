## Current state — 2026-05-25 Run-now UX paper-cut + backend wall-budget tightening (ADR-093)

**Deliverables:** ADR-093 in DECISIONS.md. UI edits in [web/app/[product]/RunNowButton.tsx](../web/app/[product]/RunNowButton.tsx) (`POLL_TIMEOUT_MS` 15 → 20 min; new `POST_DEADLINE_POLL_MS = 60_000`; `revalidateAndReload()` helper extracted; `startPostDeadlinePoll(since)` slow background poll; `pollUntilComplete` deadline-expiry rewrite). Backend edits in [worker/src/product_search/adapters/universal_ai.py](../worker/src/product_search/adapters/universal_ai.py) (`_RUN_BUDGET_SECONDS` default 600 → 480 with comment; mid-escalation budget check inside `_fetch_with_escalation`). 1 new worker test `test_fetch_escalation_bails_mid_ladder_when_budget_exceeded`.

**What shipped:**
- **UI — A1 timeout bump.** `POLL_TIMEOUT_MS = 20 * 60_000`. After Part B, worst-case backend run is ~12 min; 20-min foreground deadline leaves a 5-min cushion.
- **UI — A2 graceful post-deadline recovery.** At deadline expiry, do one final `/api/run-status` fetch + branch: `completed && success` runs `revalidateAndReload()` (rescues this exact race — the 5-seconds-early failure mode is gone); `in_progress` or `queued` switches to a setTimeout-chained 60s background poll that auto-reloads on completion, with the user-visible message `"Still running. This page will refresh when it finishes."`; `completed && !success` reports the failure conclusion (unchanged path); fetch-error falls through to the unchanged terminal `"Timed out waiting for run to complete"`. State stays `polling` during background-poll so the button remains disabled and the elapsed timer keeps ticking — no new UI surface, no new state.
- **Backend — B1 budget tightening.** `_RUN_BUDGET_SECONDS` default 600 → 480 (8 min). Env override (`UNIVERSAL_AI_RUN_BUDGET_SECONDS`) preserved. Rationale: enough for 2-4 healthy sources + AI filter + synth + commit; tight enough that a stuck tail doesn't push past the new UI 20-min deadline.
- **Backend — B2 mid-escalation budget check.** At the top of the `for i, opts in enumerate(ladder, start=1):` loop body in `_fetch_with_escalation`, `if i > 1 and _budget_exceeded(): break`. Logs a `"attempt N: SKIPPED (per-run budget exceeded)"` entry in `attempts` and a warning line. The function's existing `best` fallback returns the strongest weak body seen and lands in the `degraded=True` branch, so the source still increments the breaker counter.
- magsstore.com's ~6-min legitimate escalation in the originating run is NOT killed: the mid-escalation check only fires after the **run-wide** deadline trips, and magsstore was the very first source (the budget cannot trip mid-magsstore by definition).
- Green: worker pytest 354/354 (353 baseline + 1 new). Web tsc 0 errors; eslint 4 pre-existing warnings unchanged; `test:guards` 11/11; `test:parity` 2/2.
- Verification on the deployed app: pending after Vercel auto-redeploys post-push. Plan is to re-run the same `the-week-1yr-subscription` Run-now and confirm (a) backend completes in ≤12 min, (b) UI shows the report directly OR shows the "still running, will refresh" path if backend somehow exceeds 20 min, (c) no terminal "timed out" for any successful run.

**Finding worth remembering:** When a "timed out" UX symptom traces to a UI poll deadline that's just barely shorter than the backend's worst-case wall-clock, the fix needs **both** layers: bump the deadline AND make the deadline non-terminal. A graceful "still running, will refresh" path defends against the unbounded long tail (a new product with 12+ niche vendors could still exceed 20 min); a hard timeout on a deployed pipeline that the user already paid for is almost never the right UX. For the backend, a wall-clock budget that's only checked at source-entry is **not** a wall-clock budget; the mid-fetch check is the load-bearing piece that makes the budget actually fire. The smallest invariant — "this run will not exceed N minutes" — needs to be enforced at every yield point, not just the cheap ones.
